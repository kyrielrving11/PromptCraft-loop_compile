# Tree-of-Thought (ToT) Prompting

## Purpose

适合需要多路径探索、比较、剪枝和选择最佳方案的高风险任务。ToT 的核心不是让模型"多想一点"，而是把问题求解视为对多个候选思路的搜索：每个节点是一个可公开的中间思路或部分解；模型在每一步生成多个候选、评价候选、保留最有希望的分支、必要时剪枝或回溯，最后用最优路径或综合结果给出答案。

## On the "3 Experts" Prompt Pattern

"Imagine three different experts..." 可以体现 ToT 的一部分思想：多候选、多人视角、逐轮共享、发现错误后退出、迭代改进。它们适合作为直观的"专家协作式思维树"简化提示。

但它们还不等于完整 ToT。缺少的关键控制包括：
- 明确定义每个 `thought` 是什么粒度：一句策略、一个公式、一个计划段落、一个部分答案等。
- 限制分支数和最大深度，避免输出失控。
- 明确每轮如何评价候选：分数、等级、投票、可行/不可行、风险。
- 明确如何保留、剪枝、回溯或合并分支。
- 明确最终决策规则：最高分路径、投票获胜、综合多个优秀分支，或在冲突时使用评估标准裁决。
- 明确输出格式，尤其是表格字段和最终答案位置。

所以，生成的提示词应保留"专家协作"的可读性，同时补上 ToT 的搜索控制结构。

## Input Requirements

- **用户任务**：说明希望模型完成什么任务。
- **待处理内容**：需要推理、规划、选择、创作、排错或生成的具体问题；如果暂未提供，保留占位符。
- **输出格式**：说明最终答案的结构、字段、长度、语言、代码格式或是否只输出最终答案。
- **可选案例**：用户可以在第一次请求中直接提供 0~5 组 `任务 -> 候选分支/专家思路 -> 评估/剪枝依据 -> 最终合并结果`。
- **可选信息**：专家角色、分支数、最大深度、搜索策略、评价标准、剪枝规则、是否展示树过程。

## Recommended JSON Input

```json
{
  "task": "为复杂问题生成思维树提示词",
  "role": "你是一名严谨的多路径问题求解助手",
  "target_input": "{用户输入的问题}",
  "output_format": "先输出思维树过程表，再输出最终答案。",
  "examples": [
    {
      "task": "示例任务",
      "branches": ["候选路径 A", "候选路径 B", "候选路径 C"],
      "evaluation": "根据可行性、风险和目标一致性剪枝",
      "final": "整合最佳分支后形成最终答案"
    }
  ],
  "constraints": ["每轮最多 3 个候选", "最多 3 轮", "最终答案必须简洁"],
  "branch_count": 3,
  "max_depth": 3,
  "keep_count": 2,
  "search_strategy": "beam",
  "evaluation_criteria": ["正确性", "可行性", "约束匹配", "风险"],
  "decision_rule": "选择最高分路径；若两个分支互补，则综合后给出一个答案。"
}
```

`examples` 可省略；省略时按零样本思维树提示生成。

## Thought Unit Granularity

根据任务类型选择 `thought` 的粒度：

| 任务类型 | 思维单元定义 | 示例 |
|---------|------------|------|
| **Math / Symbol** | 一步操作、一个中间式、一个局部解 | 一个公式变形、一个中间计算结果 |
| **Writing / Creative** | 一个写作计划、一个段落方案、一个叙事方向 | 一个开头策略、一个故事转折方案 |
| **Decision / Planning** | 一个策略、一个行动方案、一个风险处理路径 | 一个部署方案、一个架构选择 |
| **Code / Debug** | 一个假设、一个诊断步骤、一个修复方案 | 一个 Bug 根因假设、一个重构方案 |

## Search Strategy Selection

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **Beam（宽度优先近似）** | 每轮生成多个候选，保留前 `b` 个继续展开 | 规划、创作、数学搜索和多数通用任务 |
| **DFS（深度优先近似）** | 优先深入最高分分支，遇到不可行或低分时回溯 | 谜题、约束满足、排错 |
| **Expert-Panel（专家协作）** | 3+ 个专家各自产生候选，再相互评价和修正 | 用户喜欢可读讨论过程的场景（仍需加入分支数、深度、评分和决策规则） |

## Method Steps

1. 要求模型生成 2-4 个候选思路或方案分支。
2. 对每个分支，按用户定义的约束评估可行性、安全性和效率。
3. 剪枝：淘汰明显不成立或有严重缺陷的分支。
4. 保留最优 1-2 个分支，深入展开。
5. 给出最终方案及其选择理由。
6. 限制分支数量和探索深度，避免搜索过程撑爆上下文。

## Design Rules

- `thought` 是可公开的中间语义单元，不是隐藏思维链。要求模型输出"候选思路/方案摘要/评估理由"，不要要求暴露完整内部推理。
- 每轮必须先生成多个候选，再评价，再保留/剪枝；不要一条路走到底。
- 输出必须有明确状态表：轮次、分支编号、候选思路、评分/判断、保留或剪枝原因。
- 分支数和深度必须有限制（默认 branch_count=3, max_depth=3, keep_count=2），防止 ToT 提示词变成无限展开。
- 专家角色可用于增强多样性，但专家不是 ToT 的必要条件；关键是分支搜索与状态评价。
- 如果任务有硬约束，评价标准必须把硬约束放在第一位。
- 如果所有分支都低质量，允许回溯到上一轮重新生成候选。
- 在输出评估与剪枝过程时保持精简。对于明显违背物理常识或分布式原理的劣质分支，可直接在内部思维链中抛弃，仅输出具备高可行性的候选分支进行对比。
- 对严格格式输出任务，树搜索过程可以内部完成，最终只输出用户指定格式。

## When to Use

- 任务存在多个可能的解决方案，需要比较权衡。
- 结果质量非常重要（安全、性能、正确性），不能只靠单一路径。
- 功能完全独立（不依赖前序上下文）且认知负载高。
- 常见场景：核心算法实现、加密/签名验证、Assembly 操作、安全审计关键路径。

## Example Use Cases

- "用纯 Assembly 手写一个防重放攻击的默克尔树签名校验函数"
- "实现一个高性能的并发哈希表，对比锁策略"
- "设计合约升级方案，评估 delegatecall vs UUPS vs 钻石标准"

## Prompt Output Template

当你用 Tree-of-Thought 技术生成提示词时，按以下 8 节骨架输出。

### 章节骨架

| # | 章节 | 必需 | 说明 |
|---|------|------|------|
| 1 | 角色 | ✓ | 多路径问题求解专家，可引入"3 位专家"角色 |
| 2 | 任务 | ✓ | 一句话 |
| 3 | 输入 | ✓ | 高风险的复杂问题 |
| 4 | 输出格式 | ✓ | "先输出思维树过程，再输出最终答案/方案" |
| 5 | **思维树探索框架** | ✓ | 见下方格式规范 |
| 6 | 具体实现要求 | ✓ | 按输出格式展开 |
| 7 | 硬约束 | ✓ | 含分支数、深度、剪枝规则、安全/性能约束 |
| 8 | 生成要求 | ✓ | "先探索多路径，再选择最优方案" |

### 第 5 节格式规范：思维树探索框架

必须包含以下三个子块：

**A. 搜索策略声明：**

```
搜索策略: [beam / dfs / expert-panel]
分支数(branch_count): 2-4
最大深度(max_depth): ≤3
每轮保留数(keep_count): 1-2
```

**B. 评估标准表：**

| 标准 | 权重 | 说明 |
|------|------|------|
| 正确性 | 最高 | 逻辑/数学正确 |
| 可行性 | 高 | 可实际编码运行 |
| 约束匹配 | 高 | 满足硬约束 |
| 性能 | 中 | 时间/空间效率 |
| 安全性 | 最高 | 无漏洞/无越权 |

**C. 思维树状态表格式（要求模型按此格式输出）：**

```
| 轮次 | 分支 | 候选方案 | 评估 | 决策 |
|------|------|---------|------|------|
| 1    | A    | [方案描述] | 评分/判断 | 保留/剪枝 |
| 1    | B    | [方案描述] | 评分/判断 | 保留/剪枝 |
| 1    | C    | [方案描述] | 评分/判断 | 保留/剪枝 |
| 2    | A1   | [深入展开] | ... | ... |
| 2    | B1   | [深入展开] | ... | ... |
| ...  | ...  | ...       | ... | ... |

最终选择: [最优方案 + 选择理由]
```

### 搜索策略选择指南

根据任务类型选择策略（详见上文 Search Strategy Selection 表）：
- **Beam（默认）**：规划、创作、数学搜索 — 每轮生成多个候选，保留前 b 个
- **DFS**：谜题、约束满足、排错 — 优先深入最高分分支，遇到不可行时回溯
- **Expert-Panel**：用户喜欢可读讨论过程的场景 — 3+ 专家各自生成候选，相互评价

### 质量自检

- [ ] 分支数 2-4，深度≤3（防止 token 爆炸）
- [ ] 评估标准表包含正确性、可行性、约束匹配三项以上
- [ ] 状态表格式明确（轮次/分支/方案/评估/决策）
- [ ] 从现有 design_rules 中选择合适的搜索策略
- [ ] 硬约束在评估标准中排在第一位
- [ ] 不是"3 位专家随便聊聊"——有明确的分支数、深度、评分和剪枝规则
- [ ] 结尾要求模型输出最终选择和理由

## Notes

- ToT 成本通常高于 CoT，因为它要生成和评价多个分支；只有复杂任务值得使用。
- 限制分支数量（2-4 分支，深度 ≤ 3 层），避免 token 爆炸。
- 剪枝标准必须在 Prompt 中明确写出。
- 不要把"专家人数"误当成 ToT 的核心。核心是：候选生成、状态评价、搜索/剪枝、最终选择。
- 与 step-back 的区别：ToT 是横向多分支并行 + 剪枝选择，step-back 是纵向抽象一个原则然后落地。
- 与 least-to-most 的区别：ToT 是探索性的（不确定最优路径），least-to-most 是确定性的（已知最佳顺序）。
- 如果用户同时有 step-back 意图和 ToT 需求，将 step-back 作为每个候选分支评估前的抽象步骤。
- 用户提供案例时，可以作为少样本参考嵌入提示词，但案例服务于分支生成、评估、剪枝和合并风格，不应削弱思维树结构。
